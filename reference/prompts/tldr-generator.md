# tldr-generator

用途：第 2.7 步 TL;DR 段落生成。用 Claude `general-purpose` sub-agent，模型指定 Opus。按占位符替换：

- `{{TARGET_DATE}}`
- `{{FINAL_MARKDOWN_PATH}}`
- `{{LAST_ERROR_SECTION}}`：首次调用为空字符串；重试时填入"[上一次产出被验证器拒绝，原因如下，请避开]\n<errors>"

```text
你的任务：为一篇已经写完的中文日报生成 TL;DR 段落。读者是日报作者自己（半年后回看）。

## 输入

读取 {{FINAL_MARKDOWN_PATH}}，这是已完成正文 + 隐私复审的 {{TARGET_DATE}} 日报全文。你只基于这份文档的内容生成 TL;DR，不读任何其他文件、session jsonl、git log 或外部资料。

## 输出格式

纯文本，不带标题、不带 bullet、不带表格、不带 markdown 修饰。只要几段 prose。

## 硬约束

1. **长度严格上限**：2–3 段，**总长不得超过 400 字**。字数按去掉首尾空白后的**全部字符数**计（含中英文、数字、标点、内部空格和换行，每个字符计 1）。下限 100 字。落笔后自己用这个口径数一遍，超过就删到合规再交。
2. **视角**：第一人称（"我"），写给半年后的自己看。
3. **主线副线结构**：整天只有 1–2 件事是主线，其余副线或只字不提。不要把所有 session 平权列出。次要的事可以完全略过。
4. **禁术语堆叠**：单句里出现 3 个及以上名词化抽象术语（如"耦合 / 根因 / 先验 / 字段 / 维度 / 闭环 / 语义 / 前提 / 判断力 / 产出结构 / 地基"等）时必须拆句或换大白话。黑话不是显得深刻，是挡路。
5. **禁止口癖与元评论**：不得出现"喵"、"哈哈"、"嗯"、"对了"等对话语气词；不得带签名、问候语；不得有自我指涉（"这份 TL;DR 说的是…"、"总结一下…"、"综上所述…"、"以上…"）。只写日报内容本身。
6. **禁止出现**：session ID（`#xxxxxxxx` 这种）、"锚点:" 字样、commit hash、具体文件路径（含扩展名的文件名、绝对路径都算）、表格、bullet 列表、heading。
7. **必须覆盖三件事**（可融成一两段，不必严格分段）：
   - 今天主要在干什么
   - 为什么这样做 / 这件事有什么意思
   - 从中学到了什么 / 明天要改什么

## 正例风格参考（不要照抄措辞，只学语感）

    昨天的工作节奏是"闷头写代码"模式——没有对话协作，两个项目各自推进。ProjectA 的核心产出是一次架构级的技术决策和一个新功能。ProjectB 则是密集的小幅优化，属于版本发布前的打磨阶段。

    技术选型不是越"高级"越好，要匹配产品定位。单文件零运维的工具比"正经"的重型方案更符合自托管场景。这个回退不是退步，是对"什么才是对的"的重新认识。

## 反例（避免）

    今天做了 A 改造、B 审查、C 接入、D 优化、E hook、F 修复、G 清理、H 安装、I 注入。反方耦合材料体积与工作重心的根因是缺活动性质先验，把机械字段与观点字段切开两端兼得。

（原因：平铺罗列无主次 + 单句 5+ 抽象术语）

{{LAST_ERROR_SECTION}}

## 交付

直接输出 TL;DR 正文本身，不要任何前言、解释、标题、尾注、签名。输出的每一个字符都会被写入日报。
```
